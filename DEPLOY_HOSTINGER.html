Deploy no VPS Hostinger — passo a passo

Este guia leva um VPS Hostinger com Ubuntu do zero até o site “Classificação de Beneficiários do Programa” no ar, com HTTPS, servindo o front estático (modelo/dist) e o backend FastAPI atrás do Nginx em /api.

Escopo atual (fatia de autenticação): este deploy já permite testar em produção login, troca de senha no 1º acesso e “esqueci minha senha”. As telas de seleção/validação ainda usam dados mock até os Blocos C–E do plano — mas a infraestrutura de deploy é a mesma, então atualizá-la depois será só git pull + restart + rebuild do front.
O que JÁ dá para testar em produção com este deployO que vem depois (Blocos C–E)
GET /api/health; login real; troca de senha no 1º acesso; “esqueci minha senha”; criação de usuário via CLI; e-mail de credenciais (se o SMTP estiver configurado). Filtro por grupo nos seletores (/api/contexto); validação real da planilha e envio do .xlsx por e-mail (/api/validar); “Baixar modelo” (/api/modelo).
Opção rápida (script automático). Em vez de rodar os passos 1–9 na mão, há um script que faz tudo: deploy_hostinger.sh (na raiz do repo). No servidor, como root, com o DNS já apontando para o VPS: 
curl -fsSL https://raw.githubusercontent.com/GiovanniCharret/sistema_gclt_demo/main/deploy_hostinger.sh -o deploy.sh
sudo DOMINIO=SEU_DOMINIO.com.br ADMIN_EMAIL=voce@exemplo.com bash deploy.sh
Ele instala tudo, gera a SECRET_KEY, sobe o serviço, configura o Nginx e o HTTPS, e ao final mostra o comando para criar o 1º usuário. Os passos abaixo continuam valendo como referência/explicação e para o diagnóstico se algo falhar.
Convenções deste guia. Substitua os placeholders: SEU_DOMINIO.com.br (seu domínio/subdomínio), IP_DO_VPS (IP do servidor) e o e-mail do admin. O projeto será instalado em /opt/anexov, rodando sob o usuário deploy.

0Pré-requisitos

  • VPS Hostinger com Ubuntu 22.04 ou 24.04 e acesso root via SSH (painel Hostinger → VPS → dados de acesso).
  • Um domínio (ou subdomínio) apontando para o IP do VPS: no hPanel/DNS, crie um registro A SEU_DOMINIO.com.br → IP_DO_VPS. HTTPS exige domínio (não funciona só com IP).
  • Acesso ao repositório github.com/GiovanniCharret/sistema_gclt_demo (se privado, um token/deploy key).
  • (Opcional, para e-mail real) credenciais SMTP de um provedor.

1Acessar e preparar o servidor

Do seu PC, conecte por SSH e atualize o sistema:

# no seu computador
ssh root@IP_DO_VPS

# já no servidor
apt update && apt -y upgrade
apt -y install git nginx curl ufw

Firewall (libera SSH e Nginx):

ufw allow OpenSSH
ufw allow 'Nginx Full'
ufw --force enable

Crie um usuário de aplicação (não rode o serviço como root):

adduser --disabled-password --gecos "" deploy
usermod -aG sudo deploy
mkdir -p /opt/anexov && chown deploy:deploy /opt/anexov

Instale o uv (gerencia o Python 3.12, igual ao ambiente de dev) para o usuário deploy:

su - deploy
curl -LsSf https://astral.sh/uv/install.sh | sh
source ~/.bashrc        # garante o uv no PATH
uv --version

2Baixar o projeto

# como usuário deploy
cd /opt/anexov
git clone https://github.com/GiovanniCharret/sistema_gclt_demo.git .
ls        # deve listar backend/  modelo/  entrada/  base_contratos.json ...
Atenção ao ponto final (.) do git clone … . — ele faz o conteúdo cair direto em /opt/anexov. Sem o ., o git cria uma subpasta (ex.: sistema_gclt_demo/) e todos os caminhos deste guia deixam de bater. Se isso acontecer, corrija: 
# ainda em /opt/anexov
ls -a                        # veja o nome exato da subpasta
rm -rf sistema_gclt_demo     # remova a subpasta (ajuste o nome)
git clone https://github.com/GiovanniCharret/sistema_gclt_demo.git .
ls                           # deve listar backend/ modelo/ entrada/ ...
(git clone … . exige a pasta vazia; por isso remova a subpasta antes.)
O entrada/ e o base_contratos.json vêm versionados no repo — o backend precisa deles. Um processo externo atualiza o entrada/ diariamente (fora do escopo deste guia).

3Backend: ambiente, dependências e .env

cd /opt/anexov
uv venv
uv pip install -r backend/requirements.txt

Gere uma SECRET_KEY forte (assina o token de sessão):

.venv/bin/python -c "import secrets; print(secrets.token_urlsafe(48))"

Crie o arquivo backend/.env (segredo — fica fora do git). Baseie-se no backend/.env.example. Mínimo para a fatia de auth:

# backend/.env
SECRET_KEY=<cole-o-valor-gerado-acima>
TOKEN_TTL=28800

# E-mail — opção A: SEM enviar ainda (dry-run). Crie o usuário com senha conhecida.
SMTP_DRYRUN=1

# E-mail — opção B: enviar a senha temporária de verdade (troque DRYRUN p/ 0):
# SMTP_DRYRUN=0
# SMTP_HOST=smtp.seu-provedor.com
# SMTP_PORT=587
# SMTP_USER=usuario
# SMTP_PASS=senha
# SMTP_FROM=nao-responder@SEU_DOMINIO.com.br
# SMTP_TLS=1

Teste rápido que o backend sobe (depois interrompa com Ctrl+C):

cd /opt/anexov
.venv/bin/python -m uvicorn backend.app:app --host 127.0.0.1 --port 8000 &
sleep 3
curl -s http://127.0.0.1:8000/api/health ; echo
kill %1
Esperado: um JSON {"status":"ok", "referencia":{...}, "integridade":{...}}.

4Rodar o backend como serviço (systemd)

Crie o serviço (como root ou sudo):

sudo tee /etc/systemd/system/anexov-api.service >/dev/null <<'EOF'
[Unit]
Description=Anexo V API (FastAPI/uvicorn)
After=network.target

[Service]
User=deploy
WorkingDirectory=/opt/anexov
ExecStart=/opt/anexov/.venv/bin/python -m uvicorn backend.app:app --host 127.0.0.1 --port 8000
Restart=always
Environment=PYTHONUNBUFFERED=1

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable --now anexov-api
sudo systemctl status anexov-api --no-pager
Crítico: o WorkingDirectory tem que ser a raiz do projeto (/opt/anexov). O backend lê o backend/.env e o backend/usuarios.json por caminho relativo à raiz — se o diretório de trabalho for outro, o login não acha os usuários. Rode o CLI de usuários (passo 8) também a partir de /opt/anexov.

5Front: gerar e publicar o dist

O build de produção já aponta a API para /api (mesma origem, atrás do Nginx) — não precisa configurar nada. Duas opções:

Opção A (recomendada): buildar no seu PC e enviar o dist

# no seu computador, na pasta do projeto
cd modelo
npm install
npm run build          # gera modelo/dist

# envia o dist para o servidor
scp -r dist/* deploy@IP_DO_VPS:/opt/anexov/modelo/dist/

Opção B: buildar no próprio servidor

# como deploy, instale Node 20+ via nvm
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
source ~/.bashrc
nvm install 20
cd /opt/anexov/modelo
npm install && npm run build

6Nginx: servir o front + proxy do /api

sudo tee /etc/nginx/sites-available/anexov >/dev/null <<'EOF'
server {
    listen 80;
    server_name SEU_DOMINIO.com.br;

    root /opt/anexov/modelo/dist;
    index index.html;

    # SPA: qualquer rota cai no index.html
    location / {
        try_files $uri $uri/ /index.html;
    }

    # API: repassa para o uvicorn mantendo o prefixo /api
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        client_max_body_size 50m;   # planilhas grandes (Blocos D/E)
    }
}
EOF

sudo ln -sf /etc/nginx/sites-available/anexov /etc/nginx/sites-enabled/anexov
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx

Teste em HTTP antes do TLS: curl -s http://SEU_DOMINIO.com.br/api/health deve responder o JSON.

7HTTPS (obrigatório — o login trafega senha)

sudo apt -y install certbot python3-certbot-nginx
sudo certbot --nginx -d SEU_DOMINIO.com.br

O certbot configura o 443 e o redirecionamento de 80→443 automaticamente, e renova sozinho. Confirme abrindo https://SEU_DOMINIO.com.br.

Não pule o HTTPS. Sem ele, e-mails/senhas trafegam em texto puro. O guard de token e o login só devem ir a produção sob TLS.

8Criar o primeiro usuário (no servidor)

Sempre a partir da raiz do projeto (mesmo diretório do serviço):

cd /opt/anexov
.venv/bin/python -m backend.admin_usuarios add fulano@distribuidora.com.br
  • Com SMTP_DRYRUN=0 (passo 3, opção B): o usuário recebe a senha temporária por e-mail e a troca no 1º acesso.
  • Com SMTP_DRYRUN=1: nada é enviado. Para testar mesmo assim, crie com uma senha conhecida:
cd /opt/anexov
.venv/bin/python -c "from backend.auth import criar_usuario; criar_usuario('fulano@distribuidora.com.br', senha='Temp123'); print('ok')"
O grupo econômico que o usuário enxerga vem do domínio do e-mail (mapa provisório em backend/acesso.py — confirmar domínios reais antes do uso amplo). Isso só afeta os seletores (Bloco C); não impede o teste de login agora.

9Testar em produção

  1. Abra https://SEU_DOMINIO.com.br.
  2. Login com o e-mail criado e a senha temporária → deve abrir “Definir nova senha”.
  3. Defina a nova senha (≥ 6 caracteres) → deve entrar (cai no Menu Principal — ainda mock).
  4. Saia e entre de novo com a nova senha → entra direto.
  5. Sanidade da API: curl -s https://SEU_DOMINIO.com.br/api/health.
Se isso funcionar, a parte sensível do deploy está validada: Nginx + uvicorn + /api + TLS + auth. Os próximos blocos reaproveitam esta mesma base.

10Operação e atualizações

Quando houver novidades no repositório (ex.: Blocos C–E):

cd /opt/anexov
git pull
uv pip install -r backend/requirements.txt   # se mudaram deps
sudo systemctl restart anexov-api            # reinicia o backend

# front: rebuild + republicar o dist (opção A do passo 5) OU buildar no servidor

Logs e diagnóstico:

journalctl -u anexov-api -f          # logs do backend em tempo real
sudo tail -f /var/log/nginx/error.log
sudo systemctl status anexov-api --no-pager
O entrada/ é recarregado sozinho quando os CSVs mudam (o backend detecta pelo mtime) — não precisa reiniciar o serviço na atualização diária.

11Checklist de segurança

  • HTTPS ativo (passo 7).
  • SECRET_KEY forte e única, só no backend/.env (nunca no git).
  • backend/usuarios.json é segredo (hashes de senha) — já é gitignored; faça backup dele à parte.
  • Serviço roda como deploy, não root.
  • Firewall (ufw) só com SSH e Nginx.
  • Se usar SMTP, guarde as credenciais só no .env.
